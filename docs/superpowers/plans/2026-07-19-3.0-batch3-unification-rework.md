**Reference implementation notes (ProvToolbox, checked 2026-07-19 at `~/projects/ProvToolbox/modules-validation/prov-validation/src/main/java/org/openprovenance/prov/validation/`):**
- **Architecture:** ProvToolbox implements PROV-CONSTRAINTS normalisation *inside its validation engine* (`Validate.java`), not as a standalone `unified()`: pipeline = `Expansion` (materialise existentials) → `Indexer` (same-id statements merged on insert via `resolveDuplicate`) → a **fixpoint loop** over `{applyUnification; uniqueAll}` driven by a `merger.updated` flag → remove unified entries, apply the final substitution → type inference → normal form. That whole shape corresponds to prov's #62 validation-engine backlog item, not to this batch's `unified()` scope.
- **Placeholder story corroborates our locked decision:** the Java object model also has no distinct `-` — absent optional arguments are `null`, and `Expansion` replaces each `null` with a **fresh gensym existential** (`VarQName`/`VarTime`, gated by `Config.DEFINITION_OPTIONAL_PLACEHOLDERS`). So the reference implementation treats absent as existential too; our "absent = existential" reading matches it exactly.
- **Merging shape matches Task 1:** `Merger.mergeGenericUnwrapped` unifies pairwise by positional argument index (id, time args, qname args), unions type/extra attributes, and fails the merge on constant-vs-constant mismatch — the same positional-over-`FORMAL_ATTRIBUTES` design used here. Difference in failure API: ProvToolbox *reports* failed merges into a `ValidationReport` (`addToFailedMerge`); `unified()` raising `ProvUnificationError` is a deliberate contract difference appropriate to a normalisation utility rather than a validator.
- **Why we don't need its variable machinery:** ProvToolbox needs materialised existentials + a global substitution table with propagation (`unificationSubstitution`, `updateSubstitutionTable`) + the fixpoint loop **because it implements Constraints 24–29** (`Uniqueness` keys generation/invalidation/start/end on endpoint-pair `TwoKeys` tables and unifies activity times against start/end events) — those merges bind identifiers and cascade. With 24–29 out of scope here, id-groups are independent and all identifiers in play are constants, so per-group positional resolution suffices; no substitution environment is required. **If #62 ever pulls 24–29 in, adopt the ProvToolbox shape (expansion + substitution fixpoint), not an extension of this batch's simpler merge.** Record this note in the #62-facing docs text (Task 4).

def test_spec_worked_example_merges():
    # PROV-CONSTRAINTS §6.1 worked example (Constraint 22).

    document = _doc()
    document.activity("ex:a", startTime=T1)
    document.activity("ex:a", startTime=T2)

        document.unified()

    unified = document.unified()
    associations = [r for r in unified.get_records() if r.identifier is not None]
    assert len(associations) == 1
